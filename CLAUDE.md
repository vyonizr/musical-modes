# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

```bash
npm run dev      # starts Vite dev server
npm run build    # production build (Vite + PWA service worker)
npm start        # serve production build via Vite preview
```

Tests: `npm test` (vitest). No linter — do not add one without an explicit request.

## Architecture

Single-page React app built with Vite. `src/App.tsx` composes the app from extracted hooks/components rather than holding all state itself: `useChordKeyboardInput` (keyboard-to-chord engine), `useOnboardingTour` (Joyride tour state), `KeyDetectorPanel` (key-detection form), and `usePlayProgression` (progression playback) each own their own state — `App.tsx` retains only `selectedScale`/`activeModes`/`preferSharp`/`volume`, since those are read by multiple children (`ProgressionsPanel`, `ProgressionBuilder`, the mode table, URL sync, embed building). `TableContent.tsx` is purely presentational.

**Core data flow:**
1. User picks a root key → `generateModes(scale, preferSharp)` in `src/utils/generateModes.ts` computes all 7 diatonic modes by rotating the Ionian interval pattern (`W W H W W W H`)
2. Returns `Mode[]` (name, intervals, romanNumerals, chords) — types in `src/utils/types/index.ts`
3. `TableContent` renders one row per mode; chord cells call `triggerAttackChord()` / `triggerReleaseChord()` in `src/utils/chords.ts` (Tone.js `PolySynth`, lazily initialised on first interaction)

**URL state:** `?key=C&modes=ionian,dorian&acc=flat` — key, active modes, and accidental preference are synced to/from the query string via `history.replaceState`. `♭` is encoded as `b` in the URL; `acc` is `flat` or `sharp`.

**Key selector UI:** Root key is chosen via a CSS piano keyboard (`.piano-keyboard`, white/black key buttons) and a `.chromatic-grid` fallback grid (visible at smaller widths). Both render from `PIANO_WHITE_KEYS`, `PIANO_BLACK_KEYS`, `CHROMATIC_ROW1`, `CHROMATIC_ROW2` in `src/utils/constants.ts` and call `setSelectedScale(KEYS[index])`. The ♭/♯ toggle (`preferSharp` state) switches which of `KEYS` / `KEYS_SHARP` is displayed.

**Keyboard mapping:** `KEY_ROWS` in `src/utils/constants.ts` maps keyboard rows (Q–U / A–J / Z–M) to chord columns for the first three active mode rows. Modifier keys change chord flavour: `,` = flat-7, `.` = maj7, `k` = sus2, `l` = sus4. Modifiers can be pressed before or after a chord key — held chords transform live. `useChordKeyboardInput(selectedScale, activeModes, preferSharp)` in `src/utils/useChordKeyboardInput.ts` owns this: it holds the `keydown`/`keyup` listeners plus five refs (mirroring `selectedScale`/`activeModes`/`preferSharp` to dodge stale closures, plus the held-flavour and held-chords refs) and returns `{ activeFlavour, keyboardPressedChords }` to `App.tsx`. `findChordForKey(key, activeModesData)` is exported separately as a pure function (no refs, no DOM) specifically so the key→chord lookup is unit-testable without a DOM — see `useChordKeyboardInput.test.tsx`, which also exercises the full hook via `@testing-library/react`'s `renderHook` under a per-file `// @vitest-environment jsdom` pragma (the only test file in this repo that needs jsdom — everything else runs in vitest's default `node` environment). Follow the same refs-alongside-state pattern if extending.

**Chord flavours:** `ChordFlavor` (`"flat7" | "maj7" | "sus4" | "sus2"`) in `src/utils/chords.ts` threads from `useChordKeyboardInput`'s `activeFlavour` return value → `App.tsx` → `TableContent` as `activeFlavour` prop → `display7thChordName()` / `displaySusChordName()` changes the displayed label; `triggerAttackChord(chord, flavour)` changes the notes played. Guitar voicing tables for each flavour/quality combination are in `chords.ts`. Every chord root is folded into the same fixed register (`ANCHOR_MIDI ± 6` semitones) before offsets are applied, so switching roots never causes an octave jump — see `docs/sdd/specs/2026-07-07-chord-voicing-register-balance.md`.

**Onboarding tour:** `useOnboardingTour(isTouchDevice)` in `src/utils/useOnboardingTour.tsx` owns all tour state (`TOUR_STEPS`, `tourRun`, `tourStepIndex`) and returns `{ visibleSteps, tourRun, tourStepIndex, handleJoyrideEvent, restartTour }` to `App.tsx`, which passes them straight to `<Joyride>`. `react-joyride` runs once on first visit; `localStorage('musical-modes-tour-seen')` gates it. Restartable via the `?` link in the footer. `TOUR_STEPS` must be computed inside the hook (not hoisted out) since it calls `t()` from `useTranslation()`, which only works during a component/hook render — this is also why the file is `.tsx` (it returns JSX for two tour steps), not `.ts` like the repo's other hooks.

**Copy / i18n:** all UI copy lives in `src/locales/en.json`, accessed via `react-i18next`'s `useTranslation()` / `t()` — `src/i18n.ts` initialises `i18next` with `en` as the only, fixed language (no switcher, no detector) and is imported once for its side effect from `main.tsx`. Parameterized copy uses i18next interpolation (`"Section {{n}} chords"` + `t('keyDetector.sectionPlaceholder', { n })`), not template literals. Two spots are deliberately *not* run through `t()`: the accidental-toggle and keyboard-mapping steps in `TOUR_STEPS`, and the `#modifier-hint` paragraph in `App.tsx` — all three interleave translated text with inline `<code>` markup, which `t()` can't express; `<Trans>` is the intended upgrade path if that's ever needed. To find copy that's no longer referenced (generic unused-export tools like `knip`/`ts-prune` can't see into `en.json`), run `npm run i18n:unused` (wraps the `i18n-unused` package) — it's a manual/local check, not wired into CI. See `docs/sdd/specs/2026-07-10-extract-ui-strings.md` and `docs/sdd/specs/2026-07-10-i18next-migration.md` for the rationale.

**Import paths** use `baseUrl: "."` (tsconfig), so imports look like `src/utils/types` not `../../utils/types`. Follow existing pattern.

**Design docs** live in `docs/sdd/specs/` as dated markdown files (e.g. `2026-07-01-feature-name.md`). Check there before implementing a feature — specs often contain rationale, voicing tables, and out-of-scope decisions that aren't obvious from the code.

**Chord progression examples:** `PROGRESSIONS` in `src/utils/progressions.ts` is a hand-authored list (`Progression { id, name, bpm?, steps: {mode, degreeIndex, flavour?, bars?}[], songs }`). Rendered by `ProgressionsPanel.tsx` (collapsible panel above the mode table), which resolves each step's roman numeral against the currently-selected key's `Mode[]` and plays it via `onPlay` → the same `triggerAttackChord`/`triggerReleaseChord` chord engine. Convention: `id` is the kebab-case of `name`, and both should describe the progression's harmonic movement/character, not just the source song. Borrowed chords (e.g. a ♭VII landing in a major-key progression) are built by pointing a step at `aeolian`'s scale degree while the rest of the progression's steps use `ionian` — see `double-plagal` and `backdoor-vamp` — rather than modeling the whole progression in `mixolydian`, to stay consistent with how `detectKey` now prefers to explain these shapes (see below). Cadence labels are *not* shown in this panel (removed — was previously computed via `classifyCadence` but read as redundant/noisy next to the roman-numeral pattern).

**Key finder:** `detectKey(sections, preferSharp)` in `src/utils/detectKey.ts` takes user-typed chord sections (space/comma separated tokens, one string per section) and scores every root × `{ionian, aeolian, dorian, mixolydian}` combination to guess the key, driven from `src/components/KeyDetectorPanel.tsx` — a self-contained panel (owns its own `open`/`sections`/`results` state, following the same `interface IProps` pattern as `ProgressionsPanel`) that takes `{ preferSharp, onKeySelected(root, modes) }` as its only props and calls `onKeySelected` once the user picks a detected key, letting `App.tsx` set `selectedScale`/`activeModes` without needing to know anything about detection internals. Scoring weights each mode's own scale-degree chords (`WEIGHTS`, tonic/IV weighted highest), adds a smaller `BORROWED_WEIGHT` for chords explainable via `BORROW_SOURCES` (e.g. ionian borrowing from aeolian/lydian), and penalizes chords no candidate mode can explain (`NON_DIATONIC_WEIGHT`). Bonuses reward opening/closing on the tonic, a IV→I or V→I resolution anywhere in a section, and the *whole progression* (not just a section) ending on the tonic. **Deliberate tie-break:** a bare `♭VII-IV-I` vamp (e.g. `G F C G`) resolves to the *Ionian* reading with `♭VII` borrowed from Aeolian, not the Mixolydian tonic reading — `BORROWED_WEIGHT` (0.75) is tuned so a borrowed-but-otherwise-fully-major explanation edges out a same-score modal one. If you touch `WEIGHTS`/`BORROWED_WEIGHT`, re-check `detectKey.test.ts`'s mixolydian-vs-ionian cases — they encode this preference on purpose, it's not an oversight.

**Cadence classification:** `classifyCadence(romanNumerals)` in `src/utils/cadence.ts` names a small set of hard-coded roman-numeral patterns (Authentic, Plagal, Double Plagal, Aeolian, Half cadence) by scanning for them anywhere in the sequence (not just at the end) and returns per-step root-motion labels (`authentic`/`plagal`/`step-up`/etc.) regardless of match. It is currently exercised only by `cadence.test.ts` / `progressions.test.ts` and is not wired into any UI — reach for it if you need cadence naming again, rather than re-deriving the pattern list.

**Embeddable progression widget:** `?embed=1` in the URL makes `main.tsx` mount `src/components/EmbedWidget.tsx` instead of `App` — same single Vite entry/bundle, no router. The embed resolves the step list from the query string in order of precedence: `?steps=` (see below) first, then `?progression=` (a `PROGRESSIONS[].id`), then falls back to the first PROGRESSIONS entry. It renders one `TableContent` row per *distinct* mode in the progression's `steps` — so a modal-mixture progression (`double-plagal`, `backdoor-vamp`) shows multiple mode rows in the embed, with the currently-playing cell highlighted the same way the main table does. Chord-sequencing playback (cancel-and-release-on-retrigger, per-step timing off `bpm`) lives in `src/utils/usePlayProgression.ts`, shared by both `App.tsx` and `EmbedWidget.tsx` — this was extracted from `App.tsx` specifically so the embed wouldn't reimplement it; don't duplicate it a third time.

**Custom step encoding:** `encodeSteps()`/`decodeSteps()` in `src/utils/steps.ts` serialize `ProgressionStep[]` to/from a URL-safe string: each step is `mode:degreeIndex[:flavour[:bars]]`, joined by `,`. Flavours omitted from the right end are dropped; bars default to 1. Example: `ionian:0:flat7:2,aeolian:2,sus4`. `?steps=...` is the primary embed input — used by the ProgressionBuilder's "Copy embed code" action. See `src/utils/steps.test.ts` for the encoding contract.

**ProgressionBuilder (`src/components/ProgressionBuilder.tsx`):** A collapsible panel below `ProgressionsPanel` that lets users build custom progressions step by step. Each step has a mode selector, degree selector (showing roman numerals for the current key), flavour dropdown, and bars input. Users can add/remove steps, set BPM, preview via the shared hook, and copy a `?steps=`-based embed snippet. No persistence — the only output is a one-off embed URL.

**`TableContent` `showKeyHint` prop:** Added to suppress the keyboard shortcut hint (`key-hint` span, normally `Q`/`A`/`Z`-style) inside each chord cell. Passed as `false` by `EmbedWidget` since keyboard mappings are meaningless in an iframe. Defaults to `true` so the main app is unaffected.

`ProgressionsPanel`'s "Copy embed code" button (`handleCopyEmbed` in `App.tsx`) calls `buildEmbedSnippet(progression, selectedScale, preferSharp, origin)` in `src/utils/embed.ts`, which builds the full `<iframe>` HTML string from the main app's *currently selected* key/accidental plus the progression's own `bpm`, sizing the iframe internally via `computeEmbedHeight(modeCount)` (still separately exported and unit-tested, since it's pure pixel math with its own edge cases — see `embed.test.ts`). `App.tsx` passes `window.location.origin` in explicitly rather than the function reading it itself, keeping `buildEmbedSnippet` pure and directly testable. See `docs/sdd/specs/2026-07-11-embeddable-progression-widget.md` and `docs/sdd/specs/2026-07-11-embed-widget-custom-progressions.md` for the full rationale.

## Key Gotchas

- **PWA / service worker only activates in production.** `vite-plugin-pwa` builds the SW via `injectManifest` strategy during `vite build`; `vite.config.ts` configures it. The SW is not registered by the browser in dev.
- **Chord cells use `onMouseDown`/`onMouseUp`/`onMouseLeave`**, not `onClick` — `onMouseDown` is intentional for mobile autoplay policy; `onMouseUp`/`onMouseLeave` trigger release. Do not change.
- **iOS audio is blocked by design.** WebAudio requires a user gesture; the app does not work around this. Known limitation, not a bug.
- **`tsconfig.json` has `strict: false`.** Do not tighten without being asked.
- **`package.json` overrides** (`serialize-javascript`, `postcss`) are security pins. Preserve them.
- **Prettier config** (`prettierrc.json`) is empty — uses defaults only.
- **Package manager**: use `npm` (`package-lock.json` is the only lockfile present).
- **`computeEmbedHeight()` (`src/utils/embed.ts`) hand-computes iframe height instead of measuring the DOM** — it hardcodes pixel offsets for the embed header, per-mode-row height, and padding. If `.embed-header`, `.mode-row`, or `.mode-name-header` rules in `modes.css` change, update `computeEmbedHeight` to match or previously-copied `<iframe>` embed snippets will clip or leave dead space (the copied snippet's height is baked in at copy time — it doesn't update itself later).
- **`BPM_DEFAULT` (120, in `progressions.ts`) has one other fallback that must stay in sync by hand:** the `?? 120` in `usePlayProgression.ts`'s duration calc. This already drifted out of sync once (fixed in commit `19ab10e`) — grep for `?? 120` if you ever change the default. The embed path's fallback no longer duplicates this by hand: `buildEmbedSnippet` in `src/utils/embed.ts` imports `BPM_DEFAULT` directly.
- **No `X-Frame-Options`/CSP `frame-ancestors` header exists in `netlify.toml`.** This is required for `?embed=1` iframes to load on external sites — don't add either header without carving out an exception for the embed route.
- **`BPM_DEFAULT` is now exported from `progressions.ts`.** Import it instead of hardcoding 120 in new code (see the builder for the pattern).
- **`@testing-library/react` and `jsdom` are devDependencies, but only one test file uses them.** `useChordKeyboardInput.test.tsx` opts into jsdom via a `// @vitest-environment jsdom` docblock at the top of the file — this does not change the environment for any other test. Every other test in the repo runs in vitest's default `node` environment (no DOM), which is faster and was sufficient before this hook existed. Don't add jsdom globally in `vite.config.ts` unless multiple files actually need it.
- **`?steps=` takes priority over `?progression=` in the embed.** If both are present, `?steps=` is used. A malformed `?steps=` (empty mode, non-numeric degree) falls through to the `?progression=`/PROGRESSIONS fallback.
